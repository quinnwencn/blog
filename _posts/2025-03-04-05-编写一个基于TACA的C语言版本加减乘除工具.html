根据这个架构，我们可以编写自己的加减乘除计算器。我们的工程，将以OP-TEE的hello_world为基础，增加自定义的功能。

  • 拷贝hello_world为coperate
    • `bash

    cd optee_example

    cp -r hello_world coperate

    `
  • 修改TEE的UUID

    • 每对TA/CA对应的UUID都是唯一的，这也是REE侧的CA能准确匹配到TEE侧对应CA的依据，因此我们需要为我们的coperate生成一个独一无二的UUID，我们可以使用OP-TEE推荐的网站生成：ITU-T UUID generator

    获取UUID后，我们需要现在hello_ta.h(原来的hello_world_ta.h)中修改:

    `C

    #define TA_HELLO_UUID \

    { 0x35249876, 0xd953, 0x4161, \

    { 0x9d, 0x4a, 0x22, 0x06, 0x1f, 0x4b, 0x70, 0x20} } // 35249876-d953-4161-9d4a-22061f4b7020

    `

    这里需要将UUID按照对应的格式拆分填写

    然后修改TA的Makefile中的BINARY的UUID为新的值：

    `C

    BINARY=8aaaf200-2450-11e4-abe2-0002a5d5c51b

    BINARY=35249876-d953-4161-9d4a-22061f4b7020

    `

    原理

    OP-TEE的架构下，CA和TA的通信是通过建立会话，会话通过TA UUID标识符来标识，因此同一个OP-TEE OS上，不同的CA/TA通过不同的TA UUID可以和匹配的TA/CA进行通信。

    REE侧的流程

    REE侧作为会话发起方，通信流程主要如下：

  • 初始化上下文 TEEC_InitializeContext
  • 打开会话 TEEC_OpenSession
  • 唤醒命令 TEEC_InvokeCommand
  • 结束会话 TEEC_CloseSession
  • 终止上下文 TEEC_FinalizeContext

    • 其中在一个会话中，可以多次唤醒命令。

    TEE侧的流程

    TEE侧也有对应的打开会话、关闭会话和唤醒命令的入口：

    TA_OpenSessionEntryPoint

    TA_CloseSessionEntryPoint

    TA_InvokeCommandEntryPoint
